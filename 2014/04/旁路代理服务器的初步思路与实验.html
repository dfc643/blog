旁路代理服务器的初步思路与实验

Network Norckon 2014-04-08 2125℃ 0评论

在常规的网络应用当中，代理服务器目前仍然是一个重要的网络角色。其主要的作用是将两台电脑进行通信而并非直接通信，即间接通信。具体用途可以用做数据加密、内网互联等等，在大型企业网络中较为常见。

代理（英语：Proxy），也称网络代理，是一种特殊的网络服务，允许一个网络终端（一般为客户端）通过这个服务与另一个网络终端（一般为服务器）进行非直接的连接。一些网关、路由器等网络设备具备网络代理功能。一般认为代理服务有利于保障网络终端的隐私或安全，防止攻击。

但是传统的代理服务器也分为几种形式，一种是网关形式，一种是服务器客户端形式。这两种形式都需要对现有的网络或者客户端进行配置，部署起来并非十分便利，而且负载会相对较大。

传统的代理服务器需要接受客户端发过来的所有数据，然后在服务器上面进行数据的判断，是否需要继续进行代理。这样的判断将会占据掉服务器的系统资源，如果用户较多，那么可能给服务器带来较大的压力。

现在在了解完传统的代理服务器之后，北极光就来说一下旁路代理。作为旁路代理服务器是不需要和网关进行挂钩的，一般情况对于客户端也不需要做配置。客户端接入网络后，自动进入代理环境，并且根据服务器的设定为客户端提供按需的代理服务。不仅仅方便了客户，同时也方便了网络部署，而且还减轻了服务器的负担。

首先是网络的拓扑结构，对于旁路代理服务器来说，和正常的网络并没有什么两样。只是需要增添一台服务器作为代理服务器罢了。我们并不需要对原有的网络拓扑结构进行任何的更改。

从拓扑图中不难看出，我们的代理服务器和其他的用户计算机的平级的设备。这种拓扑结构和传统的 C/S 模式代理服务器拓扑是一模一样的。

既然拓扑结构是一模一样的，那么我们如何实现旁路代理，而不需要在客户端上进行任何的配置呢？这就得益于 DNS 服务，大致工作原理如下。

1、「客户端」打开 dfc643.github.io 网站并向「服务器」请求DNS查询。
2、「服务器」返回代理服务器的 IP 地址 192.168.16.1 给「客户端」。
3、「客户端」浏览器向「服务器」 192.168.16.1 请求网站数据。
4、「服务器」根据「客户端」发送的 HTTP 请求头返回相应数据。
5、「客户端」获得数据并接收，整个代理过程结束。

不难看出，和传统的 C/S 模式的代理服务器相比，DNS 取代了 C/S 这一部分的工作，于是这样就变成了旁路工作模式。

但是’是不是就是万能的呢，对于架设旁路代理服务器对软件有什么样的要求呢。这自然是有要求的，代理服务器必须安装了 DNS 服务器，而且代理服务器软件必须支持反向代理。目前由于反向代理技术的一些问题导致 HTTPS 暂时不可用。后续继续研究，如果有新的进展将会在本博发布。

如何搭建旁路代理服务器，方法很简单，首先按照 C/S 代理服务器的拓扑结构，将旁路代理服务器接入现有的计算机网络当中。

然后配置好旁路代理服务器的网络部分，保证旁路代理服务器自身可以连接上互联网，然后进行软件方面的部署。需要安装的 DNS 服务与 Nginx 反向代理（若代理软件支持反向可省略）软件以及传统代理。

然后我们先配置 Nginx 反向代理服务器，如果你的代理服务器软件本身支持反向，那么可以省略这一步。至于证书先暂时借用 GoAgent CA 的证书一用，如果没有的话，可以去谷歌下载一份。接下来我们需要修改 Nginx 的配置文件，首先你需要了解一下 Nginx 的配置文件，这里就不做介绍了。

我们添加一条 Server 配置项目，其代码如下：

server {
        listen          80;
        listen          443 ssl;
        server_name     *.com *.jp *.net *.us *.tw *.org *.edu *.cn; #此处还可以添加更多按需添加

        ssl on;
        ssl_certificate      D:/goagent/CA.crt; #此处改成你的 CRT 文件位置
        ssl_certificate_key  D:/goagent/CA.key; #此处改成你的 KEY 文件位置
        keepalive_timeout    60;

        location / {
                proxy_pass  http://127.0.0.1:8087; #代理服务器地址
                proxy_set_header Host $host;
                proxy_set_header X-Real-IP $remote_addr;
                proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        }
}

然后这样 Nginx 收到客户端的请求后，会将客户端请求的域名替换成代理服务器的地址，并且添加一些自定义的请求头再返回给代理服务器。然后代理服务器再返回数据给客户端，整个反向代理过程完成。

那么 Nginx 这时候到底做了什么呢，让我们来了解一下：

1、客户端请求网址并查询 DNS ：http://dfc643.github.io/index.htm。
2、服务器返回 DNS 查询请求 A 记录：192.168.16.1。
3、客户端向服务器发送 HTTP 请求头，其中 HOST 值为 dfc643.github.io。
4、服务器 Nginx 收到请求并判断 HOST 是否在代理范围。
5、Nginx 将用户请求的 URL 主机替换成 http://127.0.0.1:8087/index.htm。
6、Nginx 将替换过的 URL 并添加 HOST 值与其他自定义头传送给代理软件。
7、代理软件收到来自 Nginx 的数据并根据 HOST 值分析数据。
8、代理软件将得到的数据发送给客户端浏览器。
9、整个旁路代理过程完成。

可以看出 Nginx 担当了一个中转的角色，为不支持反向代理的软件提供了反向支持，如果代理软件本身就支持反向代理那么这一步可以省略。接下来就是配置代理服务器软件，此处不做介绍。

整个旁路代理过程，网关并没有掺入其中，直接由客户端与旁路代理服务器沟通，而且客户端并没有做出什么配置与更改。唯一要做的则是需要在 DHCP 服务器中将 DNS 设置成旁路代理服务器的 DNS。

至于 HTTPS 的反向代理将会继续研究，如果有进展将会立即发布文章。
