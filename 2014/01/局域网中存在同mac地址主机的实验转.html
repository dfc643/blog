局域网中存在同Mac地址主机的实验[转]

Network Norckon 2014-01-31 995℃ 0评论

网络数据从高层下来后，到达网络层，封装为IP数据包，在这一层的传输，只认识IP地址。再到达数据链路层的时候，封装为数据帧，在这一层，只认识MAC地址。在交换环境中，是靠着MAC来传输数据的。

交换机的工作原理，大家也都知道，每一个交换机的端口都有学习功能，可以记住端口所连接NIC的MAC地址，然后有数据传输需求的时候，就会智能的建立专用传输通道，分隔了冲突域，也在一定程度上保证数据安全。

可是改成同一个MAC地址，那将怎么访问呢？于是做了一个这样的实验。在局域网中存在3台这样的计算机：A(192.168.30.1)、B(192.168.30.2)、C(192.168.30.3)，其中将A、B的Mac地址修改成一样。这个时候当然A是访问不到B的，因为A访问B的数据，在交换机那里，已经不能精确的利用世界唯一的MAC来判断目标端口了。因为他们的地址是一样的，这让我想起了思科的单臂路由。

但这仅仅是假设，如果要确定，A发出的数据是立即向与B具有同一个Mac的自己转回来，还是向所有具有相同于B的Mac的端口（包括自己）转发数据帧，最好利用嗅探来确定。而且交换机的说明中绝对不会告诉你，交换碰到一样的Mac怎么处理。毕竟交换机的产生基础就是唯一的Mac地址。总之，A和B之间已经无法正常的进行通信了。

结论一：局域网一旦出现了相同的MAC的计算机，它们之间不可以进行互相通信。

既然A和B的互相通信的情况不是那么乐观，那么我们现在换个角度来看看C访问另外两台计算机的情况。计算机C的MAC地址和A、B的不一样，下面我们分别使用C来Ping计算机A与计算机B。

下面是计算机C与计算机A的Ping通信结果：丢包率较大
Pinging 192.168.30.1 with 32 bytes of data:
Request timed out.
Request timed out.
Request timed out.
Reply from 210.72.28.253: bytes=32 time<10ms TTL=64

下面是计算机C与计算机B的Ping通信结果：丢包率较大
Pinging 192.168.30.2 with 32 bytes of data:
Request timed out.
Request timed out.
Request timed out.
Request timed out.

结论二：局域网中没有修改MAC并保持唯一MAC的机器，在访问修改成重复MAC的机器时候，产生大量随机性问题。

既然A和B不能互访了，那能不能访问C呢？A虽然和B现在用了同样的MAC地址，在交换环境中，显然和传输规则背道而驰，但是本次测试作为实验，我们还需要继续往下测试。测试结果很让人意外，A、B两台计算机分别Ping计算机C都无一丢包，测试十分完美。

结论三：局域网中修改成为相同MAC的机器，访问没有修改MAC的机器时候，没有受到影响。

这些情况在以前我都没有碰到过，毕竟一般情况下局域网中间不会存在MAC地址重复的情况。但是作为实验，我们需要制造一个特殊的环境。那么我们接下来就测试A、B客户机访问互联网的情况了，测试结果非常出乎意料。

曾经有人说，局域网存在相同MAC的计算机，他们都可以上网。我绝对否定，刚才的例子证明了，修改成为同样的MAC地址，是影响交换机工作的。其中结论二非常有力说明了这一点。

但是在实际测试中，我们用计算机A打开浏览器输入网址，网站非常成功的载入了。紧接着使用计算机B输入地址，网速和平时一样的快。而作为本例中没有修改过的计算机C是应该没问题的，毕竟是世界唯一MAC的计算机。随后我们使用计算机C做了测试，果然没问题，很快的打开了网页。

其中计算机A与计算机B，它们具有同样的MAC，并且是在一个局域网中，它们竟然可以同时访问互联网，而且几乎察觉不出停滞就打开了不同的网页。不管在测试过程中更换多少个URL，同样都能够很快的打开。而对于其他软件，如聊天软件也可以正常使用。

这项实验看似推翻了前几项实验的结论二，“局域网中没有修改MAC并保持唯一MAC的机器，在访问修改成重复MAC的机器时候，产生大量随机性障碍”。但是怎么可能局域网网关在把Web服务器返回的结果传给A和B的时候，就似乎没有障碍了呢？我们先留下一个悬念，原因后面继续分析。

结论四：具有相同的MAC地址的计算机，在局域网中可以同时连接互联网。

如果你知道是怎么回事，当然不觉得惊讶。可是我却非常惊讶，我不知道为什么网关在把INTERNET上的数据请求结果发回给计算机A和B时而不出错，而实验二中的表现结果那么差劲？于是我做了如下测试。

首先我用计算机来Ping计算机A，结果显然很不好。正如结论二所述，ICMP消息返回的很少，只是偶尔出有一个返回信息。就在这个时候这个时候，在计算机A上Ping计算机C，此时计算机C上的每一条请求都得到了反馈结果，全部都可以Ping通。但是Ping计算机B没有Ping通。这个时候我在计算机B上使用了同样的方法Ping计算机C，再使用计算机C回Ping计算机B的时候，又全部Ping通。（我刚才用的PING都加了-t参数，这样可以不断的PING）

然后为了研究清楚，现在我将计算机A的Ping进程终止掉。此时此刻，计算机C在Ping计算机A的结果发生了变化，Ping到A的返回出现了大量的超时。反复重复了几次测试中得出如下的一条结论：

结论五：当修改成统一的MAC机器，在向没有更改MAC的机器提出通讯要求的时候，交换机记住了提出要求的修改成统一MAC的机器的所在端口，然后没有更改MAC的机器，可以把结果数据通过交换机的那个端口发送给请求的机器。

这就是为什么结论四，局域网网关可以把INTERNET服务器返回的数据返回给局域网中修改过MAC的机器的原因。在知道其真正的原因后自然也就觉得正常了，同时也对交换机有了新的认识。

然后我想恢复我的网络，不需要重复的MAC了，因为这样局域网操作是不能使用的。首先我把设定的静态IP删除掉，但是很快就从DHCP服务器上获得了新的IP地址，而且计算机A、B都是一样的IP地址。这是因为MAC都是一样的，DHCP给主机分配IP的原理就是每个MAC分配一个不同的IP导致。

但是我发现他们被赋予同样的IP的时候，并没有IP冲突的提示，这倒是在我意料之外的。高兴之余，我没有忘记顺手测试他们的上网功能，但是测试结果也是非常意外，他们竟然都能上网。

原来同时具有相同的MAC和IP的计算机，发出上网请求的时候，依然是交换机记住了对应的端口，然后在有数据返回的时候，按照原端口在发回去，最终计算机还是收到了数据，信息传递是很快的，数据发出到返回瞬间就完成了！

结论六：具有同样MAC的机器设置同样的IP地址，不会出现IP地址冲突错误，而且依然可以上网。

之所以这么做，一方面是为了完成这一项测试，另一方面是也只是为了娱乐，同时也知道了一些平时不知道的事情，比如局域网中机器可以用同样的IP同样的MAC来上互联网。

这个消息很好，大家都知道ADSL宽带，以前可以随意连接小型家庭局域网共享上网。可是后来网络尖兵这个设备让人困惑，他阻止私自共享上网。网络尖兵具有过滤和探测功能，他可以捕获流过的数据包中是否含有多种IP和MAC地址，可以探测用户有没有安装代理和NAT设备，以此判断是否有共享网络的存在。我们可以修改统一的MAC配置统一的IP再连接ADSL上网，网络尖兵也就无能为力了。

后记：以上的情况都是在统一的MAC和统一的IP机器数量少的情况下做的。如果机器很多的话，上网是会出现问题的。
因为很有可能，交换机在给一个机器返回数据结果的当中。有其他的统一的MAC和统一的IP机器发出上网请求，这样的话，交换机就不知道这个数据包该给那个端口返回了。因为他们具有同样的MAC地址，交换机也不好判断将数据包发送给哪一台计算机了。不过我们的应用一般都是以TCP为主，发现丢包的话计算机还再次向服务器再次进行一次请求，所以稍微延迟一下只要不超时，还是可以维持网络应用的。
